# VakayGo Security & Correctness Audit — 2026-06-25

Full multi-agent audit of the VakayGo platform (Caribbean travel marketplace —
Next.js 16 App Router, Neon Postgres + Drizzle, jose JWT auth, Stripe Connect,
124 API routes). Discovery fanned out 12 dimensions (auth, IDOR/authz, money,
business logic, injection, SSRF, cron-auth, AI abuse, info-disclosure,
input/DoS, CSRF, client/SSR exposure); each finding was adversarially verified
by two independent skeptics. 32 raw findings survived verification → 28 unique
after dedup.

**Status: all 28 fixed in code. `tsc` clean, production build green.**
Adversarially diff-reviewed before deploy.

---

## 🔴 CRITICAL (2)

### 1. Passwordless account takeover via `/api/auth/guest`
`POST /api/auth/guest {email}` looked up any existing user by email and minted
that user's session — no password, OTP, or email-ownership proof. Knowing an
admin/operator/traveler email was enough to obtain their full session
(`requireAdmin`'s DB role check passes because the session legitimately belongs
to the victim's userId).
**Fix:** the guest flow now 409s if the email already exists (forcing real
sign-in) and only mints a session for a brand-new guest user.
`app/api/auth/guest/route.ts`

### 2. Privilege escalation via signup role self-assignment
`POST /api/auth/signup` passed the client-supplied `role` straight into
`createUser`. The `"traveler"|"operator"` TypeScript type is erased at runtime
and the `user_role` enum includes `admin`, so anyone could self-register as
`admin`. Since `requireAdmin` trusts the DB role, that's full unauthenticated
admin creation.
**Fix:** signup allowlists role to `traveler`/`operator`; `createUser` coerces
defensively (never persists `admin`).
`app/api/auth/signup/route.ts`, `server/auth.ts`

## 🟠 HIGH (9)

3. **TOTP 2FA not enforced at login** — `signin` never checked TOTP, so
   enabling 2FA gave no protection. **Fix:** `verifyCredentials` returns
   `totpEnabled`/`totpSecret`; `signin` requires a valid code before issuing a
   session; the signin page now shows a 2FA step (`requires2fa`).
   `app/api/auth/signin/route.ts`, `server/auth.ts`, `app/auth/signin/page.tsx`

4. **Review rating manipulation (no booking ownership)** — reviews POST trusted
   client `bookingId`/`listingId` with no ownership/status check and allowed
   unlimited reviews. **Fix:** require the booking to belong to the reviewer +
   be `completed`, derive `listingId` from the booking, dedupe one-per-booking,
   set `isVerifiedPurchase`. `app/api/reviews/route.ts`

5. **Unauth booking-verify leaks PII + check-in token** — the public verify GET
   returned `verificationToken` and guest email for any enumerated booking id.
   **Fix:** GET now requires a matching `?token=` (constant-time) and returns
   neither the token nor the email; verify page updated.
   `app/api/bookings/[bookingId]/verify/route.ts`, `app/(platform)/verify/...`

6. **Live Resend API key hardcoded in 3 git-tracked files** — fallback secret
   in `contact`, `disputes/[id]`, and a script. **Fix:** removed all fallbacks
   (env-only). ⚠️ **Key must be rotated** (still in git history).

7. **Stored XSS via JSON-LD** — operator-controlled `listing.title`/`parish`
   were `JSON.stringify`'d straight into `<script type="application/ld+json">`,
   allowing `</script>` breakout on public SEO pages. **Fix:** new
   `lib/json-ld.ts#safeJsonLd` escapes `<`,`>`,`&`; applied to all 6
   data-driven JSON-LD pages.

8. **Operator self-publish bypass** — listings update let an operator set
   `status='active'`, skipping `pending_review` moderation. **Fix:** operators
   may only toggle an already-approved listing or submit for review; promotion
   to `active` from draft/pending/rejected requires admin.
   `app/api/listings/[slug]/update/route.ts`

9. **Unpaid booking → forced payout** — an operator could PATCH a never-paid
   booking straight to `completed`, which escrow-release + payouts then turned
   into a real payout-ledger credit. **Fix:** `completed`/`confirmed`
   transitions require `paymentId && paidAt`; both crons additionally filter
   `paidAt IS NOT NULL`. `app/api/bookings/[bookingId]/route.ts`, both crons

10. **Gift card funded before payment** — cards were created `isActive:true`
    with full balance before the PaymentIntent was captured, and no webhook
    confirmed/voided them. **Fix:** cards created `isActive:false`; the
    `payment_intent.succeeded` webhook (metadata `type=gift_card`) activates
    them. `app/api/payments/gift-cards/route.ts`, `app/api/payments/webhook/route.ts`

11. **iCal import SSRF** — operator-supplied feed URL fetched server-side
    (import + weekly cron) with no host/IP allowlist, redirect, or size cap →
    SSRF to internal services / cloud metadata. **Fix:** new `lib/safe-fetch.ts`
    blocks private/loopback/link-local/metadata/IPv4-mapped-IPv6 ranges,
    re-validates each redirect hop, caps the body at 5 MB.

## 🟡 MEDIUM (14)

12. **TTS unauth cost abuse** — `/api/tts` called paid OpenAI TTS with no auth.
    **Fix:** `requireUser`.
13. **Public paid-LLM endpoints** — `review-summary` gated behind `requireUser`
    (no public caller); `smart-search`/`search-summary` (public explore) bounded
    with input caps + `max_tokens`; `review-intelligence` left public but is
    per-listing cached + rate-limited.
14. **Unauth Claude tool loop in `/api/chat`** — bounded message count (≤40) and
    total input (≤24 000 chars).
15. **Rate-limit XFF spoof** — limiter keyed on the client-spoofable leftmost
    `X-Forwarded-For`. **Fix:** prefer platform `x-real-ip`/`x-vercel-forwarded-for`,
    else rightmost XFF. `lib/rate-limit.ts`
16. **Operator AI on any listing** — `optimize-listing`/`pricing-suggestions`
    ran paid AI on arbitrary `listingId`. **Fix:** `assertListingOwnership`.
17. **`generate-image`/`generate-blog` billable to any user** — **Fix:**
    restricted to operator/admin + prompt length caps.
18. **Cron secret unset bypass** — `enrich-listings`, `refresh-photos`,
    `migrate-photos-to-blob` lacked the `!cronSecret` null-guard (a
    `Bearer undefined` bypass if the env var is unset). **Fix:** added the guard.
19. **Google API key committed in lighthouse report** — removed the 3 tracked
    lighthouse artifacts + `.gitignore`d them. ⚠️ **Key must be rotated.**
20. **Contact form: email amplification + HTML injection** — unauth endpoint
    sent a branded email to an attacker-chosen address with unescaped fields.
    **Fix:** HTML-escape all fields, length caps, and the
    submitter-confirmation email was removed (anti-relay).
21. **Loyalty points multi-award** — points re-awarded on every `completed`
    PATCH. **Fix:** idempotent per booking + only on the transition.
22. **Cancel promised a refund it never issued** — and then blocked the refund
    route. **Fix:** cancel now issues the policy-correct refund via the shared
    `lib/cancellation.ts` engine.
23. **Payout cron non-atomic** — could write a phantom inflated payout row under
    concurrent runs. **Fix:** insert placeholder → claim with `RETURNING` →
    recompute from claimed rows → delete if zero claimed.
24. **Gift-card redeem race** — non-atomic read-modify-write double-spend.
    **Fix:** single atomic conditional `UPDATE ... WHERE balance >= amount`.
25. **Refund didn't reverse operator transfer** — on Connect destination
    charges the platform refunded the traveler but the operator kept their
    share. **Fix:** `reverse_transfer`/`refund_application_fee` when the
    operator has a connected account.

## Adversarial self-review round (caught + fixed before deploy)

A 7-agent adversarial review of the fix diff found 5 high + 9 medium + 5 low
issues in the fixes themselves; the real ones were all fixed:

- 🔴 **signin echoed the TOTP secret** — `verifyCredentials` now returns
  `totpSecret`, and signin returned `{ user }` wholesale, leaking the second
  factor. Now stripped before responding.
- **Self-publish bypass round 2** — the status guard blocked only direct
  `→active`; an operator could go `draft→paused→active`. Now every operator
  transition's source state is constrained (live states only from approved).
- **Missed XSS sink** — `components/ui/breadcrumbs.tsx` still raw-`JSON.stringify`'d
  `listing.title` into JSON-LD. Now uses `safeJsonLd`.
- **Refund `reverse_transfer` from current operator state** — would 500 (no
  refund) for operators who onboarded to Connect after a platform charge.
  `refundBooking` now detects the charge mode from the actual PaymentIntent.
- **Double refund / charge-mode** — `refundBooking` now takes a per-booking
  Stripe idempotency key so concurrent cancel/refund collapse to one refund.
- **`charge.refunded` webhook** marked partial refunds as fully `refunded`; now
  only on a full refund (`refunded===true` / `amount_refunded>=amount`).
- **Payout cron underpayment-on-crash** — amount is now computed UPFRONT so a
  mid-run failure can never leave a $0.00 ledger row.
- **Loyalty double-award race** — the `→completed` transition is now an atomic
  conditional UPDATE; points award exactly once.
- **tts regression** — reverted the `requireUser` (the concierge that calls it
  is anonymous-capable); bounded instead via the `ai` rate class + length cap.
- Hardened: IPv4-mapped-IPv6 hex form in the SSRF guard; reviews dup-race now
  returns 409 not 500; guest "account exists" now shows a sign-in CTA.

## ⚠️ Recommended follow-up (not shipped this round)
- **Returning-guest recovery (magic link).** Closing the guest-ATO hole means a
  returning customer whose email already has a (random-password) guest account
  must sign in — but no password-reset / magic-link flow exists yet. Add an
  email one-time-login link so returning guests can re-enter. (Currently they
  get a "please sign in" CTA.)

## Deferred by design (documented, not bugs)
- In-memory per-instance rate limiter (acceptable for single-region; XFF spoof
  now fixed so the per-IP limit actually binds).
- Stateless JWT without server-side revocation.
- `review-intelligence` / `smart-search` / `search-summary` remain reachable by
  anonymous visitors (public explore/listing UX) but are now cost-bounded
  (input caps, `max_tokens`, per-listing caching, hardened rate limit). Hard
  auth-gating is a one-line change if desired.
