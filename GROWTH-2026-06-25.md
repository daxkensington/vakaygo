# VakayGo Growth / SEO Diagnostic — 2026-06-25

Multi-agent diagnostic of why VakayGo gets ~0 organic traffic. The booking
machinery is genuinely strong; the problem is **indexation**, not the product.

## The core finding

`site:vakaygo.com` returns **1 result** (the homepage) despite a **7,225-URL**
sitemap and crawl-friendly robots. Mechanical cause:

- **/explore** (the priority-0.9 discovery surface) is a `"use client"` SPA —
  **zero** crawlable listing links in its SSR HTML.
- **All 18 island hubs** rendered their listings inside a client component and
  pointed category cards at the /explore SPA → **crawl dead-ends**.
- Only ~6–8 islands had SSR programmatic pages (`things-to-do-in-X` etc.), so
  **~6,500 of 7,100 listings were sitemap-only orphans** — exactly what a young,
  low-authority domain refuses to index ("Discovered – currently not indexed").

## ✅ Shipped this session (commit 8510bb2)

- **Island hub `/[island]` now SSR-renders a "Browse all N experiences" index**
  linking every active listing → every listing is reachable in **2 clicks** from
  the homepage. Removes orphan status for the ~6,500 listings (the highest-leverage

  first paint on the booking page. (`components/listings/photo-gallery.tsx`)

## ⚠️ Owner actions (cannot be fixed in code — these are the real unlock)

1. **GSC: verify vakaygo.com**, submit `sitemap.xml`, and URL-Inspect →
   Request Indexing the homepage, the 21 hubs, and top programmatic pages to
   seed crawl now that the internal links exist. (vakaygo is NOT currently in
   the GSC account.) Add Bing Webmaster Tools too.
2. **Watch GSC Page Indexing** — most listings should currently sit in
   "Discovered/Crawled – currently not indexed"; confirm that count migrates to
   "Indexed" over the weeks after this deploy.
3. **Backlinks + time** — a zero-authority domain in the Booking/Viator/
   TripAdvisor niche indexes gradually no matter what. A handful of real
   referring domains + several weeks of crawl history are required.
4. **Set realistic targets** — pursue **long-tail only** (specific operator +
   island, "things to do in <smaller island>"), not OTA-owned head terms. The
   near-term KPI is *indexation*, not rankings.

## Conversion / trust (diagnostic flagged — mostly data/policy, needs Ian)

- **Demo listings show fabricated "5.0 (22 reviews)"** while real (scraped)
  listings show "Unclaimed, 0 reviews." The JSON-LD aggregateRating is correctly
  gated to `reviewCount > 0` in code, so this is a **data** problem — decide the
  demo-listing policy: clearly label them "Sample" + strip fake ratings, or drop
  them from the indexable sitemap. (Review-spam manual-action risk.)
- **Drive real operators to CLAIM listings** (the flow exists) so pages stop
  reading "Unclaimed" — the only durable conversion fix.
- The orphan `/book` route is a hard sign-in wall that contradicts the in-page
  guest checkout — consider deleting it in favour of the widget.

## Deferred (bigger, lower urgency than the above)

- Convert **/explore to SSR** (server shell + client filter) so the priority-0.9
  surface is also a crawl hub. The island-hub browse-all already makes every
  listing crawlable, so this is an enhancement, not a blocker.
- Split `sitemap.xml` into a sitemap index with per-island child sitemaps for
  cleaner GSC per-section coverage reporting.
